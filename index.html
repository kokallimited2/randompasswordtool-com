security

    
crypto.getRandomValues() vs Math.random()

    
This matters more than most developers realise. The wrong RNG makes generated passwords predictable.

    

      

        
Math.random() UNSAFE

        
JavaScript's Math.random() uses the Mersenne Twister PRNG (or V8's xorshift128+). Neither is cryptographically secure.

        
    
          
  • ✗Internal state recoverable after 624 sequential outputs (MT)
    • 
          
  • ✗Period of 2¹⁹⁹³⁷−1 — looks random but is deterministic
    • 
          
  • ✗Not seeded from OS entropy — seeded from system clock
    • 
          
  • ✗Violates NIST SP 800-90A, RFC 4086, OWASP ASVS v4.2
    • 
          
  • ✗V8 implementation details differ between Chrome versions
    • 
        

        
// NEVER do this for passwords
function badGen(len) {
  const chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabc…';
  return Array(len).fill('')
    .map(() => chars[Math.floor(
      Math.random() * chars.length // ← INSECURE
    )]).join('');
}

      

      

        
crypto.getRandomValues() NIST-COMPLIANT

        
The Web Crypto API calls the OS-level CSPRNG directly: /dev/urandom on Linux/macOS, CryptGenRandom on Windows.

        
    
          
  • ✓State cannot be recovered from observed outputs
    • 
          
  • ✓Seeded from OS entropy pool (hardware RNG where available)
    • 
          
  • ✓Consistent behaviour across all V8/SpiderMonkey versions
    • 
          
  • ✓Compliant: NIST SP 800-90A, RFC 4086, FIPS 140-2
    • 
          
  • ✓Available in all modern browsers and Node.js ≥ 15
    • 
        

        
// Correct implementation
function secureGen(charset, len) {
  const buf = new Uint32Array(len);
  crypto.getRandomValues(buf); // ← OS entropy
  return [...buf]
    .map(n => charset[n % charset.length])
    .join('');
}

JavaScript's Math.random() uses the Mersenne Twister PRNG — its 19,937-bit internal state can be fully recovered after observing 624 consecutive outputs. This makes generated passwords statistically predictable. Use crypto.getRandomValues(), which reads from the OS-level CSPRNG (/dev/urandom on Linux, CryptGenRandom on Windows) and cannot be predicted from observed outputs.

      
The Web Crypto API's crypto.getRandomValues(), which is consistent with NIST SP 800-90A (Recommendation for Random Number Generation Using Deterministic Random Bit Generators) and RFC 4086 (Randomness Requirements for Security). It is the only browser-native random source appropriate for cryptographic use.
