From Dev to Prod: Secure Password Generation Across Environments

15 Apr 2026 · 6 min read · Alex Rivera, Staff Security Engineer

The 12-factor app methodology states credentials should come from the environment, not the codebase. In practice, this means different things at each stage: local development, CI/CD, staging, and production all have different threat models and operational constraints. This guide maps the correct pattern for each.

Local Development

Local development credentials need to be easy to set up, distinct from production, and never committed to version control. Use .env files with unique generated values per developer.

# .env.example — committed to version control
DB_PASSWORD=# generate: openssl rand -base64 24
API_SECRET=# generate: openssl rand -hex 32
REDIS_PASSWORD=# generate: openssl rand -base64 16

# .env — NOT committed (in .gitignore)
DB_PASSWORD=xK9mP2@local...
API_SECRET=a1b2c3d4e5f6...
#!/bin/bash — setup.sh: onboard a new developer
cp .env.example .env
sed -i "s|# generate: openssl rand -base64 24|$(openssl rand -base64 24)|" .env
sed -i "s|# generate: openssl rand -hex 32|$(openssl rand -hex 32)|" .env
echo "Generated fresh local credentials in .env"

Staging Environment

Staging should mirror production infrastructure but use distinct credentials. Staging credentials should be rotated on every deployment to prevent credential reuse between environments. Never use production credentials in staging — OWASP lists this as a top configuration risk.

# Dockerfile / docker-compose.yml — staging
services:
  app:
    environment:
      # Injected at runtime by CI/CD — not hardcoded
      DB_PASSWORD: ${STAGING_DB_PASSWORD}
      API_SECRET: ${STAGING_API_SECRET}
    env_file:
      - .env.staging  # never committed; injected by CI

Production: Secret Rotation on Deploy

The most mature pattern rotates credentials on every production deployment. New credential is generated, service restarts with it, old credential is invalidated. No static secret survives beyond one deploy cycle.

# deploy.sh — rotate secrets on every deploy
set -euo pipefail

# Generate new credentials
NEW_DB_PASS=$(openssl rand -base64 32 | tr -d '=/+')
echo "::add-mask::$NEW_DB_PASS"

# Rotate in database
psql "$DB_ADMIN_URL" -c "ALTER USER app PASSWORD '$NEW_DB_PASS';"

# Update secrets manager
aws secretsmanager put-secret-value   --secret-id prod/app/db-password   --secret-string "$NEW_DB_PASS"

# Deploy with new credential
kubectl rollout restart deployment/app

# Old credential is now invalid — no cleanup needed
12-factor compliance: Factor III (Config) requires all configuration that varies between deployments to be stored in environment variables — not in code, not in config files checked into version control. Generated credentials are configuration — they must be injected at runtime, not baked into images or containers.

Credential Parity Audit

Run a monthly audit that verifies: no production credentials exist in any non-production environment; all credentials in all environments were generated (not chosen); all credentials were rotated within the past 90 days (for PCI-DSS environments) or 365 days (for ISO 27001); and no credentials appear in any log stream, including application logs, CI/CD logs, and infrastructure event logs.

  • Use git-secrets or detect-secrets in pre-commit hooks to prevent credential commits
  • Scan all existing git history with trufflehog on a quarterly basis
  • Set up GitGuardian or GitHub Advanced Security secret scanning on all repositories
