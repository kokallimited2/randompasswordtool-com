Rate Limiting Credential Endpoints: Patterns and Implementation

20 Apr 2026 · 7 min read · Alex Rivera, Staff Security Engineer

Password generation and validation endpoints are high-value targets. An unprotected generation endpoint is a trivial source of credentials for credential stuffing attacks. A validation endpoint without rate limiting is open to brute force. Rate limiting is not optional for any authentication-adjacent API.

Token Bucket vs Fixed Window

The two most common rate limiting algorithms are the fixed window counter and the token bucket. Fixed window is simple but leaks at window boundaries — a client can make 100 requests at 23:59:59 and another 100 at 00:00:01, exceeding the intended limit by 2x. The token bucket refills at a constant rate and is burst-friendly without the boundary leak.

Express + express-rate-limit + Redis

// rateLimit.js
const rateLimit = require('express-rate-limit');
const RedisStore = require('rate-limit-redis');
const redis = require('redis');

const client = redis.createClient({ url: process.env.REDIS_URL });
await client.connect();

const apiLimiter = rateLimit({
  windowMs: 60 * 1000, // 1 minute
  max: 100,            // 100 req/min per IP
  standardHeaders: true, // X-RateLimit-* headers
  legacyHeaders: false,
  store: new RedisStore({ sendCommand: (...a) => client.sendCommand(a) }),
  keyGenerator: (req) => req.headers['x-api-key'] || req.ip,
  handler: (req, res) => {
    res.status(429).json({
      error: 'Too Many Requests',
      retry_after: Math.ceil(res.getHeader('Retry-After')),
    });
  },
});

app.use('/api/v1/generate', apiLimiter);

nginx Rate Limiting

For infrastructure-level protection before requests reach your application, configure nginx's limit_req module. This operates at the reverse proxy layer and prevents volume attacks from overwhelming Node.js even when the application is under load.

# nginx.conf
limit_req_zone $http_x_api_key zone=api_keys:10m rate=100r/m;
limit_req_zone $binary_remote_addr zone=ips:10m rate=20r/m;

server {
  location /api/v1/generate {
    limit_req zone=api_keys burst=20 nodelay;
    limit_req zone=ips burst=5;
    limit_req_status 429;
    proxy_pass http://node_upstream;
  }
}

What to Include in Rate Limit Responses

RFC 6585 defines 429 Too Many Requests. Include these headers to allow clients to implement backoff correctly:

HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit:     100
X-RateLimit-Remaining: 0
X-RateLimit-Reset:     1746633600  # Unix timestamp of window reset
Retry-After:           47           # Seconds until client may retry
Monitoring: Alert when any API key hits 50% of its daily rate limit before noon. Investigate immediately — this pattern indicates either a runaway automation job or a compromised key being used for bulk credential generation by an attacker.
