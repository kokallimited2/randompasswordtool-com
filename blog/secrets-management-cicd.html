Secrets Management in CI/CD: How to Stop Hardcoding Credentials

30 Apr 2026 · 8 min read · Alex Rivera, Staff Security Engineer

GitGuardian's 2024 report found that 94% of developers have hardcoded credentials at some point, and detected over 12.8 million secrets in public GitHub repositories in a single year. The pattern is always the same: a developer needs a quick credential for a test or deployment, pastes it inline, forgets it is there, and commits. This guide covers the patterns that eliminate the root cause.

The Problem with Static Secrets

Static secrets in CI/CD pipelines create a compound vulnerability. They sit in version control history, pipeline logs, artifact archives, and environment variable dumps — all of which have much weaker access controls than secrets managers. When a pipeline runs, any step that prints environment variables leaks every credential in scope.

Never do this: DATABASE_URL=postgres://admin:mysecret123@prod-db:5432/app in any file that is committed to version control, regardless of the repository's visibility setting.

GitHub Actions: The Correct Pattern

GitHub Actions secrets are encrypted at rest and masked in logs. They are the correct first-line solution for CI/CD credential management. For credentials that rotate on every deployment, generate at runtime and inject — never store the generated value as a secret itself.

# .github/workflows/deploy.yml
name: Deploy with fresh credentials

on: [push]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Generate deployment credential
        run: |
          # Generate using /dev/urandom — no external API needed
          DB_PASS=$(cat /dev/urandom | tr -dc 'A-Za-z0-9!@#$' | head -c 32)
          # Mask immediately before any logging can occur
          echo "::add-mask::$DB_PASS"
          echo "DB_PASS=$DB_PASS" >> $GITHUB_ENV

      - name: Rotate database password
        env:
          DB_ADMIN_PASS: ${{ secrets.DB_ADMIN_PASS }}
        run: |
          # DB_PASS is now masked in all subsequent log output
          psql "postgres://admin:$DB_ADMIN_PASS@$DB_HOST/app"             -c "ALTER USER app_user PASSWORD '$DB_PASS';"

HashiCorp Vault Dynamic Credentials

For production-grade pipelines, Vault's dynamic secrets eliminate static credentials entirely. Vault generates a database credential on-demand with a short TTL — it expires automatically and is never stored anywhere.

# Generate a short-lived Vault database credential
vault read database/creds/my-role

# Output:
# Key                Value
# lease_id           database/creds/my-role/AbCd…
# lease_duration     1h
# username           v-root-my-role-AbCd…
# password           A1b2C3d4…  ← generated, TTL-bounded, never stored
Best practice: Use Vault's AppRole authentication in CI/CD. The pipeline authenticates with a role_id and secret_id (rotated frequently), receives a short-lived Vault token, and uses that to fetch dynamic credentials. The Vault token itself expires before the next pipeline run.

Audit Trail Requirements

Effective secrets management requires an audit trail that answers: who requested which credential, when, from which pipeline, and for which deployment. Vault's audit logging provides this. For AWS Secrets Manager, CloudTrail captures all API calls including GetSecretValue events with the caller identity.

  • Log all credential generation events with pipeline ID and commit SHA
  • Alert on credential access outside normal pipeline windows
  • Rotate all credentials that may have been exposed in a pipeline log leak immediately
  • Use OIDC-based authentication between CI/CD systems and cloud providers — eliminates long-lived API keys entirely
